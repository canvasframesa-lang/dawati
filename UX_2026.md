# UX Patterns · 2026

> What "modern" looks like for a Saudi-facing invitation SaaS in late 2026. These are commitments, not nice-to-haves — they're table stakes for a premium product at this price point.

---

## 1. Core UX principles

1. **Mobile-first, always.** 90% of Saudi users will configure their card on a phone, in WhatsApp's in-app browser, with one thumb. Every screen passes the "one-handed Riyadh-on-the-go" test or it doesn't ship.
2. **Live preview, not blind forms.** The card preview is visible *while* the user types — not behind a "Preview" button. Use a split view on desktop, a sticky mini-preview at the top on mobile, and a tap-anywhere on the preview to focus the relevant form field.
3. **Smart defaults > more options.** If the customer picks "wedding", the verse is auto-set to Ar-Rum 21, the dua is "بارك الله لهما...", the palette is the validated gold/cosmos. They override only if they want. Default to the best.
4. **No accounts to start.** First card configuration runs anonymously (saved to localStorage + a server-side draft keyed by an opaque token). Account creation happens only at checkout. This cuts the funnel-drop dramatically.
5. **Auto-save everything.** Drafts persist for 30 days. Customer returns from a phone call → exactly where they left off. Show a subtle "محفوظ تلقائيًا · قبل ١٢ ثانية" indicator.
6. **Single-screen wherever possible.** No multi-page "wizards" with Next/Back. Use a single long-form with collapsible sections, or a swipeable mobile-sheet pattern. The user always sees scope: "أنت في ٤ من ٧".
7. **Optimistic UI.** RSVP submitted → button immediately morphs into the confirmation, then we sync to the Worker in the background. If it fails, surface a retry button — don't roll back the visual confirmation.
8. **Respect attention.** Three notifications max in any flow. No toasts for routine successes.

## 2. 2026-specific tech to leverage

| Feature | Browser support (late 2026) | Where we use it |
|---|---|---|
| **View Transitions API** | Stable in Chromium + Safari + Firefox | Page transitions in the configurator and dashboard; element-level transitions when the preview updates |
| **CSS Container Queries** | Universal | Card component adapts to its container (preview iframe at 320px shows mobile layout; main page shows desktop) |
| **CSS `@scope`** | Stable in Chromium, fallback for Safari | Scope card styles so they don't leak into the chrome |
| **`<dialog>` element** | Universal | Native modals for "are you sure?" / preview lightbox |
| **Native lazy-loading** | Universal | `loading="lazy"` on any below-the-fold imagery |
| **Speculation Rules API** | Chromium | Pre-render likely-next pages (`/configure` from `/`, `/checkout` from `/configure`) for instant nav |
| **`navigator.share()`** | iOS Safari + Android | "شارك على واتس" → native share sheet on mobile |
| **PWA + Web Share Target** | Universal | Install as app on home screen; receive shared text/images |
| **Service Worker** | Universal | Offline preview of customer's own card after first view |
| **Web Animations API** | Universal | Programmatic celebrations (cleaner than CSS keyframe spaghetti for one-shots) |
| **Intersection Observer v2** | Most | Track which card sections the user actually read (analytics, A/B optimization) |
| **CSS `color-mix()`** | Universal | Derive tints on the fly without writing variable explosions |
| **`text-wrap: balance`** | Stable Chromium + Firefox | Headlines that don't orphan the last word |
| **CSS Subgrid** | Universal | Detail rows align icon-label-value perfectly across cards |
| **HTTP/3 + Early Hints** | Cloudflare + Vercel default | Faster first paint without code changes |
| **Anchor positioning** (CSS) | Chromium | Tooltips and popovers anchored to their trigger without JS |

What we are **not** using:
- Server-sent React Server Actions for the configurator — RSC for the page shell, but form state is client-side for sub-100ms preview updates.
- AI-generated artwork on cards — quality bar isn't there yet for Arabic calligraphy + Islamic ornament. Will revisit.
- Voice input — the names + dates are too specific for voice to win on phone keyboards.

## 3. The configurator (most-used screen)

### Layout
- **Desktop:** 50/50 split. Form on the right (Arabic reads RTL → form anchors right), preview on the left. Resizable divider.
- **Mobile:** Preview sticky at top (40vh, scales with scroll), form scrolls beneath. Tap preview → expands fullscreen with a close button.

### Sections (collapsible accordion, all expanded by default)
1. **نوع المناسبة** — radio with cards (زواج · خطوبة · عيد · عقيقة · تخرّج · ...)
2. **العائلتان / المُضيف** — toggles between "couple", "single host", "two families" → conditional fields below
3. **الأسماء** — groom name, bride name, family names; toggle "إظهار اسم العروس في نص الدعوة" (default off)
4. **التاريخ والمكان** — Hijri picker (default), Gregorian auto-computed; venue with optional Google Maps link
5. **النص الديني** — verse dropdown from curated library; dua dropdown; opt-out toggle for each
6. **التخصيص** — palette swap (gold default; rose-gold, emerald, midnight options), sender voice, RSVP on/off
7. **لوحة التحكم** — admin password set/regenerate, expiry date

### Live preview behaviors
- Update debounced at 200ms (no per-keystroke flashing)
- View Transition between major template variants (smooth fade-cross between palettes)
- Tap on any text in the preview → scrolls form to that field, focuses it, highlights briefly
- Tap on a section heading in the preview → collapses that section (for "what would this look like without the verse?")
- Long-press → opens a contextual sheet: "أخفي هذا القسم" / "خصّص"

### Save states
- "محفوظ تلقائيًا" badge in the top-right with last-saved timestamp
- "غير محفوظ" amber dot if there's an in-flight save error
- "نشر الدعوة ←" sticky bottom CTA after section 1 is filled

### Mobile-specific
- Form fields use native HTML5 input types (`type="date"`, `inputmode="text"`, etc.) so the keyboard adapts
- Date picker uses the device's native one (Saudis prefer the OS Hijri picker on Saudi-config phones)
- All fields full-width with 16px font (no zoom-on-focus on iOS)
- Sticky preview can be dragged down to expand or up to minimize
- Bottom safe-area padding for iPhone

## 4. The card view (what guests see)

### First load
- Service worker pre-caches all assets on first visit
- Subsequent visits load from cache instantly, then revalidate
- Card content (SVG + minimal HTML + inline CSS) is ~30KB gzipped — feels instant

### Interactions (already designed in Nuwairi reference)
- Tap anywhere → gold ripple + petal burst from the touch point
- Hover/tap names → name glows, particles emit
- Scroll-journey milestones (verse, names, RSVP, closing)
- RSVP yes → fireworks + 110 rose-petal explosion
- RSVP no → soft acknowledgment with dua

### Additions for 2026 polish
- **"Add to calendar" chip** under the date — generates an .ics file with the venue location. Apple Calendar / Google Calendar dropdown.
- **"Map / directions" button** — opens native maps app with the venue's lat/lng.
- **"Share with family" sheet** — native share with pre-formatted Arabic message: "تشرّفوا بحضور زفاف [...] [link]"
- **Reduced-motion respect** — `@media (prefers-reduced-motion: reduce)` already in the Nuwairi CSS; ensure all canvas animations also check `matchMedia` and slow/stop
- **Color-blind safe yes/no badges** — green + ✓ for yes, red + ✕ for no (icons carry the signal, not just color)
- **Print stylesheet** — `@media print` renders a clean black-on-white version for guests who want to print

## 5. The customer dashboard

### Information density: minimal, RSVPs as the headline
- Top: 3 big stat tiles (Total / يحضر / معتذر) — same as the Nuwairi admin
- Below: the RSVP feed, newest first
- Side: a "share kit" panel (link + QR + WhatsApp deep link + download OG image)

### 2026 additions
- **Real-time updates** via Supabase Realtime → new RSVPs slide in with a soft "ding" (no sound) + a brief gold glow
- **RSVP timeline** view (toggle from list) — when did people respond? Helps the host see the share-rate spike
- **Smart insights** — "٧٢٪ من ضيوفك أكدوا الحضور — أعلى من المتوسط". Light touch, no AI hallucinations.
- **Export** — CSV + Excel + PDF guest list. One click.
- **"رسالة شكر جماعية"** — compose a thank-you to all attendees, send as a WhatsApp deep link template
- **Per-RSVP reply** — tap a guest's row → opens WhatsApp with their name pre-filled (if they shared phone)

## 6. Accessibility (WCAG 2.2 AA targets)

- All interactive elements reachable via keyboard (Tab, Enter, Esc)
- Focus rings always visible (no `outline: none` without a replacement)
- Color contrast ≥ 4.5:1 for body text, 3:1 for large text — even on the translucent glass (validated)
- All canvas animations have an `aria-hidden="true"` and a semantic equivalent (the card content is plain text underneath)
- Form errors announced via `role="alert"`, not just color
- Screen-reader-friendly labels in Arabic (the `dir="rtl"` correctly inverts focus order)
- Reduced-motion respected throughout (`prefers-reduced-motion`)
- Reduced-data respected (`prefers-reduced-data`) — skips the petals canvas, keeps the static beauty
- Touch targets ≥ 44×44px (Apple HIG)
- No traps in modals; Escape always closes

## 7. Privacy & trust signals

- **No third-party trackers.** No Google Analytics, no Facebook Pixel. Use Plausible (self-hostable) or Vercel Analytics (privacy-preserving).
- **No cookie banner.** Because no tracking cookies. Liberating.
- **Privacy page in Arabic.** Plain language, not a wall of legalese: "ما نجمع، وما لا نجمع، وكيف نحمي بيانات ضيوفك".
- **RSVP data ownership** clearly stated: "بيانات ضيوفك ملكك أنت — تقدر تحذفها أو تصدّرها أي وقت".
- **PDPL compliance** (Saudi Personal Data Protection Law): show clear consent on RSVP form, allow deletion on request, anonymize IPs after 30 days.

## 9. Anti-patterns we explicitly reject

- Auto-playing background music
- Infinite scroll on the dashboard
- Modal nesting (a modal inside a modal)
- Pulling-down-to-refresh as the only way to refresh (always have a button too)
- Asking for permissions (notifications, geolocation) on first load
- Carousels on the landing page (low engagement, accessibility nightmare)
- Skeleton loaders that lie about content shape
- Email collection before showing value
- A/B testing the pricing without clear win criteria
